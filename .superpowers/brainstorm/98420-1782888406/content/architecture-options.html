工程架构：首版速度与长期扩展怎么平衡？

点击一个方案。这里比较的是代码组织与部署边界，不是页面外观。

A

模块化单体（推荐）

一个前端应用 + 一个 BFF/API 服务，但内部按画布、组件、数据源、项目四个领域拆包。开发快，边界清晰，未来可拆服务。

React 编辑器→ BFF / API→ 统一业务接口 + PostgreSQL
B

前端重型 + 极薄后端

解析、字段推断、配置迁移等多数逻辑放浏览器，后端只存 JSON。原型最快，但数据量变大后性能、安全与版本兼容压力集中到前端。

大型 SPA→ JSON CRUD
C

微前端 + 微服务

画布、图表、数据连接器分别部署。适合多团队并行和强隔离，但基础设施、联调与发布成本远超当前内部 MVP。

画布 MF 图表 MF→ 多服务集群

A 的优势

  • 4–6 周可形成可用 MVP
  • 领域边界明确，测试容易
  • 避免过早引入分布式复杂度

A 的约束

  • 团队必须守住模块依赖规则
  • 数据查询需统一经过服务端
  • 后续拆服务要保持契约稳定
